# OctoAcme Project Management Docs

## Purpose

This README is the entry point for OctoAcme project management documentation. It provides a quick overview of how projects are run and links to detailed process guides.

## OctoAcme Project Management Process Summary

OctoAcme runs projects through a lightweight lifecycle: **Initiation → Planning → Execution → Release → Retrospective**. Work is delivered iteratively with clear ownership across Project Managers, Product Managers, Developers, QA, and Stakeholders.

### Core Roles

| Role | Responsibility |
|------|---------------|
| Project Manager (PM) | Coordinates delivery, schedules, risk, and communications |
| Product Manager (PdM) | Defines outcomes, prioritizes the backlog, and measures success |
| Developers | Implement features, collaborate on design and testability |
| QA / Testing | Validate quality and acceptance criteria |
| Stakeholders | Provide inputs and approvals |

### Team Cadence / Execution Rhythm

- Twice-weekly standups for the delivery team
- Weekly sync between PM and PdM
- Monthly stakeholder updates
- Sprint/iteration planning and review at the start and end of each cycle
- Ad-hoc escalations as needed

### Risk and Communication Approach

Risks and dependencies are tracked in a **Risk Register** maintained by the PM throughout the project. The team uses a structured communication cadence to surface blockers early, escalate when needed, and keep stakeholders informed with regular status updates.

### Release Practices

Releases follow a gated process: features must meet the **Definition of Done** (including QA sign-off and CI checks) before deployment. Each release includes deployment verification and a team announcement to stakeholders.

### Continuous Improvement Loop

After every release the team holds a retrospective to identify what went well, what could improve, and what actions to take. Action items are logged and tracked to closure, closing the feedback loop and incrementally improving team processes.
